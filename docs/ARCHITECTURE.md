# 架构说明

## 当前状态

项目尚无应用代码，技术栈和部署方式均未确定。本文件只记录当前能从产品范围推导出的系统边界与约束，不代表具体实现设计。

## 系统边界

Textora 是运行在 macOS 上的桌面应用，核心职责是读取、呈现、编辑并保存用户选择的文本文件。代码、Markdown 和 Mermaid 是文本内容之上的能力，不应改变原始文本作为可编辑数据源的地位。

## 概念模块

以下是职责边界，不要求一开始对应独立类型、包或进程：

- **应用外壳**：窗口、菜单、命令、设置和生命周期。
- **文档与文件访问**：打开、读取、保存、另存、外部变更和错误处理。
- **编辑器核心**：文本模型、选择、撤销/重做和编辑交互。
- **语言与格式能力**：代码、Markdown、Mermaid 的识别、增强和预览。
- **用户状态**：仅保存确有需要的偏好与可恢复会话状态。

## 基本数据流

```text
用户选择文件 → 文件访问 → 文本文档 → 编辑器
                                 ├→ 代码/Markdown 增强
                                 └→ Mermaid/Markdown 预览
编辑结果 → 文本文档 → 文件访问 → 用户文件
```

## 架构不变量

- 文本文档是编辑与保存的权威数据源；预览或分析结果是可重建的派生数据。
- 保存操作必须明确处理失败，不能在失败时把未保存状态误报为已保存。
- 未明确获准时，不应静默改变文件编码、换行符或内容语义。
- 格式增强或预览失败不应阻止用户继续查看、编辑或保存原始文本。
- 面向特定格式的能力应与通用文件 I/O 和基础编辑职责保持边界。

## 开放问题

- 使用 SwiftUI、AppKit、混合方案还是其他 UI 技术？
- 文本编辑器采用系统文本组件、自研内核还是第三方组件？
- 文档模型与窗口/标签模型如何对应？
- 语法解析和高亮采用何种方案？
- Markdown 与 Mermaid 在本地如何安全渲染，是否允许脚本或网络访问？
- 大文件、字符编码、外部文件变更和崩溃恢复的目标与策略是什么？
- 应用沙盒、权限模型、最低 macOS 版本和分发渠道是什么？

这些问题应在首个相关功能进入规格或实现前决定；未决定前不要将候选方案写成架构事实。
