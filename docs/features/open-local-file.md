# 打开本地文本文件

> 状态：已确认

## 背景与目标

Textora 当前只能新建并编辑内存文档，已经完成的 Rust 文档核心尚未接入桌面文件选择和前端会话。本功能交付基础文本编辑规格中的下一个最小垂直切片：用户可通过系统文件对话框选择一个受支持的本地文本文件，并在不破坏当前编辑内容的前提下查看和继续编辑。

## 范围

- 提供清晰可发现的“打开”入口，并使用系统文件对话框选择单个文件。
- 打开不超过 50 MiB 的 UTF-8、UTF-8 BOM 或严格 GBK/Windows CP936 文本文件。
- 使用 Rust 文档核心建立一致快照，向前端提供 Unicode 内容及文档描述信息。
- 打开成功后更新编辑器、文件名、编码、换行类型和修改状态。
- 处理加载、取消、读取失败、文件过大、无效编码和读取期间变更。
- 当前文档有未保存修改时，在替换内容前提供明确的放弃修改确认。

## 非范围

- 保存、另存为、编码转换、混合换行选择和原子替换。
- 多标签、多窗口、最近文件、目录浏览、拖放打开和批量打开。
- 外部修改持续监听、自动重新加载和保存冲突处理。
- GB18030、其他传统编码、超过 50 MiB 文件或用户手动指定编码。
- 文件菜单和键盘快捷键；本切片只要求一个可发现的打开入口。

## 用户流程

1. 用户触发“打开”。
2. 若当前文档已修改，应用要求确认放弃修改；取消时保持原文档。
3. 应用显示系统文件对话框；用户取消时保持原文档。
4. 用户选择文件后，应用显示加载状态并由 Rust 文档核心读取和识别文件。
5. 打开成功后，编辑器显示文件内容，标签与状态栏显示文件名、编码和换行类型，文档处于未修改状态。
6. 打开失败时，应用显示可理解的错误，原文档内容和修改状态不变。

## 行为规则与边界情况

- 文件选择、读取和会话替换必须作为对用户原子可见的流程；成功前不得清空或覆盖当前编辑内容。
- 同一窗口同时只允许一个打开流程；加载期间禁用重复触发。
- 系统对话框取消不是错误，不显示错误提示，也不改变当前文档。
- 未保存确认只提供“放弃修改并继续”和“取消”；保存功能完成前不得提供无法兑现的“保存后打开”。
- 50 MiB 文件允许打开，超过限制时明确拒绝；失败不得使前端持有部分内容。
- 自动检测顺序保持为 UTF-8 BOM、严格 UTF-8、严格 CP936；纯 ASCII 按 UTF-8。
- UTF-8 BOM 不显示为编辑内容，但必须保留在编码元数据中。
- LF、CRLF 与 Mixed 必须按 Rust 文档核心结果显示，不在本切片中转换。
- 文件在读取期间变化时明确失败，用户可重新发起打开操作。
- 错误提示应区分文件过大、编码不支持或无效、读取期间变化和一般读取失败，不向用户暴露 Rust 内部调试文本。
- 打开成功后的编辑行为与内存新文档一致；首次修改后标签和状态栏必须显示未保存状态。
- 前端不得因此获得宽泛文件系统、shell 或网络权限。
- 文本内容必须通过二进制 IPC 传输，不得编码为 JSON 数字数组或大型 JSON 字符串。

## 验收条件

- [ ] 用户可以通过系统文件对话框选择并打开一个本地文件。
- [ ] ASCII、UTF-8、UTF-8 BOM 和合法 CP936 文件内容正确显示，文件名、编码和换行信息正确。
- [ ] 打开成功后文档为未修改状态，继续输入后内容和未保存状态正常更新，编辑器焦点不会因会话同步丢失。
- [ ] 用户取消文件对话框时，当前文档内容和状态不变。
- [ ] 当前文档已修改时，取消放弃确认会保留内容；确认后才允许后续文件选择和替换。
- [ ] 非法编码、GB18030 四字节、超过 50 MiB、读取失败或读取期间变化均被明确拒绝，当前文档不变。
- [ ] 加载期间不能重复启动打开流程，完成或失败后入口恢复可用。
- [ ] 大文本内容经二进制 IPC 传输，前端未获得宽泛文件系统权限。
- [ ] macOS 13+ 与 Windows 10 22H2+ / Windows 11 的行为设计一致；无法在当前环境验证的平台必须明确记录。

## 依赖与约束

- 属于 `docs/features/basic-text-editing.md` 的子切片。
- 遵守 `docs/ARCHITECTURE.md` 的 Rust 文档核心、受限权限、文档权威源和二进制 IPC 不变量。
- 编码与文件大小边界由 D-005 约束；跨平台行为由 D-003、D-004 约束。
- 复用现有 Rust `open_document(&Path)`，不得在前端重复实现编码检测、大小限制或文件读取。
