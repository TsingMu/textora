# Rust 文档编码与安全保存核心

> 状态：已完成

## 背景与目标

Textora 已能通过 Rust 文档核心安全打开受支持的本地文本文件，但编辑后的内容尚不能写回磁盘。保存是可能覆盖用户数据的高风险边界，必须先在 Rust 内建立可独立验证的严格编码、换行处理、外部修改检测和原子替换能力，再接入 IPC 与界面。

本功能交付保存流程的底层最小切片：给定已打开文档的路径、原始文件指纹、目标编码、目标换行类型和 Unicode 文本，Rust 文档核心要么完整保存并返回新的文档描述信息，要么明确失败且不破坏原文件。

## 范围

- 将 Unicode 文本严格编码为 UTF-8、UTF-8 BOM 或 GBK/Windows CP936。
- 将编辑器文本规范化为指定的 LF 或 CRLF 换行。
- 普通保存默认沿用已打开文档的编码；UTF-8 BOM 必须按原标记写回。
- 写入前校验当前磁盘文件与打开时指纹，发现缺失或内容变化时报告保存冲突。
- 编码后的文件超过 50 MiB 时在写入前拒绝保存。
- 在目标文件所在目录完成临时写入和原子替换，成功后返回新的字节数、指纹、编码和换行信息。
- 定义可供后续 IPC 稳定映射的保存错误类型，并通过真实临时文件测试关键失败路径。

## 非范围

- Tauri 保存命令、前端二进制请求、保存按钮、菜单或键盘快捷键。
- 系统“另存为”对话框、新建文档首次保存和显式编码转换界面。
- 重新加载、强制覆盖或取消等保存冲突交互。
- 自动保存、备份文件、版本历史、崩溃恢复和持续外部修改监听。
- 扩展属性、资源叉、所有者与 ACL 的跨平台保留策略；本切片不得主动扩大目标文件权限。
- 保留 Mixed 换行的逐行分布；本切片只接受明确的 LF 或 CRLF 保存目标。
- GB18030、其他传统编码、超过 50 MiB 文件和十六进制编辑。

## 用户流程

1. 后续上层保存流程提交当前 Unicode 文本、目标路径、打开时指纹、目标编码和 LF/CRLF 目标。
2. Rust 核心先规范化换行并执行无替换编码，随后检查输出大小。
3. Rust 核心确认磁盘目标仍与打开时指纹一致且允许写入。
4. Rust 核心在同目录写入临时文件，完整落盘后原子替换目标。
5. 成功时返回更新后的描述信息；任何失败都保留原文件并由上层保持未保存状态。

## 行为规则与边界情况

- UTF-8 无 BOM 保存不得添加编码 BOM；UTF-8 BOM 保存必须由编码层添加且只添加一个 BOM，文本内容本身的 U+FEFF 不得被删改。
- CP936 编码不得插入替代字符。遇到无法表示的 Unicode 字符时，保存必须在写入前失败，并至少报告首个失败字符及其在输入中的 UTF-8 字节偏移，供上层形成可理解提示。
- CP936/UTF-8 编码歧义：只有目标 GBK 字节经本工程打开流程仍识别为 GBK、且还原为相同内容时才允许普通保存；否则核心以独立 `EncodingAmbiguous` 错误拒绝，并提示另存为 UTF-8。这既包括重开会得到不同内容的情况（如「一」的 `D2 BB` 会被读成 U+04BB），也包括内容相同但会丢失 GBK 编码身份的纯 ASCII 与空内容。详见 D-006。
- 换行规范化必须把编辑器逻辑换行统一转换为目标 LF 或 CRLF，不得改变换行以外的文本内容。
- 原文档为 Mixed 时，调用方未明确提供 LF 或 CRLF 目标必须失败；核心不得自行选择。
- 编码后恰好 50 MiB 允许保存，超过 50 MiB 必须在创建或覆盖目标内容前拒绝。
- 保存路径若为符号链接，核心必须解析到可靠的真实目标，对真实目标做冲突检测、临时写入与原子替换；不得用 `rename` 替换链接目录项（否则链接被删、原目标未更新）。
- 保存前磁盘文件缺失或原始字节指纹变化都视为冲突，不得静默重建或覆盖；内容完全相同的文件身份变化不单独视为冲突。**冲突检测为 best-effort**：核心在初次校验后、原子替换前再次校验以尽量缩小窗口，但再次校验与 `rename` 之间仍存在狭窄 TOCTOU 窗口；跨平台没有通用的「按版本/身份原子替换」原语（强制文件锁不可移植且对非协作进程无效），本切片不声称竞态已严格关闭。
- 描述信息标记为只读时必须拒绝普通保存。核心在保存前与最终替换前各做一次只读重检，并在替换前把**当前**磁盘权限（含 `0600` 与执行位）应用到临时文件。**只读与权限保护为 best-effort**：最终只读检查、权限设置与 `rename` 之间仍存在狭窄 TOCTOU 窗口，目标可能在最终检查通过后、`rename` 之前变为只读而被覆盖（原子 rename 在 Unix 上是目录操作，不要求目标可写）；跨平台无严格原语，本切片不声称保存期间变只读一定会被拒绝，也不声称权限一定不被改变。
- 临时文件必须位于目标文件所在目录，以保证替换不跨文件系统；名称不得与目标或既有用户文件冲突。
- 写入、刷新、关闭或替换任一步失败时，原文件内容必须保持不变；可清理的临时文件不得残留。
- 只有替换成功后才能返回成功。成功结果中的指纹和字节数必须基于最终写入的原始字节重新计算。
- 保存错误必须区分不可编码字符、编码歧义、Mixed 未选择换行、大小超限、只读、外部修改冲突和一般 I/O 失败，不向上层暴露内部临时路径或调试文本。
- 核心接口只接收路径和字节/文本数据，不扩大前端文件系统、shell 或网络权限。

## 验收条件

- [x] UTF-8、UTF-8 BOM 和合法 CP936 内容保存后可由现有 `open_document` 重新打开，内容、编码和换行信息正确。
- [x] UTF-8 BOM 不会丢失或重复；无 BOM UTF-8 不会被静默添加 BOM。
- [x] LF 与 CRLF 按目标写入；Mixed 未指定 LF/CRLF 时明确失败且原文件不变。
- [x] CP936 无法表示的字符在写入前被拒绝，错误包含首个失败字符和位置，原文件不变。
- [x] CP936 字节经 `open_document` 后不能同时保持 GBK 编码身份与相同内容时，以独立 `EncodingAmbiguous` 错误被拒绝；覆盖「一」重开为不同内容，以及纯 ASCII/空内容重开为 UTF-8。
- [x] 编码后 50 MiB 允许保存，超过限制时拒绝且原文件不变。
- [x] 文件缺失或内容已外部修改时报告冲突，不覆盖磁盘变化（best-effort：缩小窗口，残留 TOCTOU 已记录）。
- [x] 只读文档在保存前被拒绝；保存期间被改为只读通常会被最终重检拒绝，模拟的写入/刷新/替换失败不损坏原文件、不误报成功（只读与权限保护为 best-effort，残留 TOCTOU 已记录）。
- [x] 通过符号链接保存时链接保留、真实目标内容被原子更新。
- [x] 成功保存采用同目录原子替换，返回的字节数与指纹和最终文件一致，且不残留临时文件。
- [x] 保存错误可稳定分类（含编码歧义）且不泄露内部临时路径。
- [x] macOS 与 Windows 使用同一核心行为；当前环境无法验证的平台必须明确记录。

## 验证记录

实现、审查修复与自动化验证已完成（2026-07-21）。本切片为纯 Rust 核心，未接入 IPC 与界面，各验收条件由确定性单元/集成测试覆盖；Windows 与 Clippy 因当前环境不可用而保留为待验证项。

审查修复要点：

- **收紧 CP936 往返判定**：`encode_cp936` 用 `reopens_as_same_encoding_and_content(bytes, content)` 同时核对 `detect_and_decode` 返回的编码与内容。纯 ASCII 与空内容会按 UTF-8 重开，无法保持 GBK 编码身份，因而与「一」这类重开内容变化一起以 `EncodingAmbiguous` 拒绝；字节并非合法 UTF-8、且能按 GBK 完整还原的内容（如「一 中」混排）仍可正常保存。该限制已同步到 D-006 与基础文本编辑规格。
- **符号链接策略**：`save_document_with_before_replace` 先用 `std::fs::canonicalize` 把保存路径解析为真实目标（跟随符号链接），再对真实目标做冲突检测、临时写入与原子替换。`rename` 不再作用于链接目录项，因而链接保留、真实目标内容更新。
- **只读/权限保护明确为 best-effort**：保留「`rename` 前再次 `capture_writable_permissions`」缩小窗口，但最终只读检查、权限设置与 `rename` 之间仍存在狭窄 TOCTOU 窗口（目标可能在最终检查通过后、`rename` 之前变只读而被覆盖；Unix 上原子 rename 是目录操作，不要求目标可写）。规格、验收与代码注释据此降级，不声称保存期间变只读一定被拒绝、也不声称权限一定不被改变。
- **README**：`safe-save-core.md` 由「当前规格」改为「已完成」，并更新「当前状态」段对保存进度的描述。

第二轮修复（仍有效）：CP936 可表示性用「`encoding_rs::GBK` 无替换编码 + `validate_cp936_structure` 严格帧校验」判定（不经 UTF-8 优先的 `detect_and_decode`，避免把「一」误报为不可编码）；`rename` 前再次校验冲突为 best-effort；`document/test_support.rs` 的 `TestDir`（PID + 纳秒 + 序号 + RAII）消除跨 `cargo test` 进程的目录撞名。
