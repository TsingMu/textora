# 保存已打开文件

> 状态：已确认

## 背景与目标

Textora 已能打开并编辑本地文本文件，Rust 文档核心也已具备严格编码、换行规范化、冲突检测和原子替换能力，但两者尚未通过 IPC 与前端会话连接。本功能交付下一个最小垂直切片：用户可以把当前已打开文件的修改安全保存回原路径，并在失败时继续保有编辑内容和未保存状态。

## 范围

- 为当前已打开且已修改的文件提供清晰可发现的普通保存入口。
- 通过受限 Tauri 命令把当前 Unicode 内容交给 Rust 保存核心，沿用打开时的路径、编码、换行、指纹和只读信息。
- 文本内容使用二进制 IPC 传输，不进入大型 JSON 字符串或数字数组。
- 保存成功后更新文件指纹、字节数及会话状态，并清除未保存标记。
- 映射并展示大小超限、只读、Mixed 未选择换行、不可编码、编码歧义、外部修改冲突和一般 I/O 失败。
- 保存期间协调编辑、打开和重复保存操作，避免成功结果覆盖更新的会话状态。

## 非范围

- 新建文档首次保存、另存为、改名、移动文件和显式编码转换。
- Mixed 换行的 LF/CRLF 选择界面。
- 冲突后的重新加载、强制覆盖或取消选择；本切片只安全拒绝并说明原因。
- 关闭未保存文档保护、多标签、多窗口、自动保存、备份和持续外部修改监听。
- 文件菜单、系统菜单和键盘快捷键；本切片只要求一个可发现的保存入口。

## 用户流程

1. 用户打开本地文件并修改内容。
2. 用户触发“保存”，应用进入保存状态并阻止并发文件操作。
3. 前端以文档 ID 和二进制 UTF-8 内容请求后端保存；后端从可信会话状态取得路径及打开时元数据。
4. Rust 保存核心完成编码、校验和原子替换。
5. 成功时应用更新描述信息、清除未保存状态并继续显示当前内容；失败时显示可理解的错误，内容和未保存状态保持不变。

## 行为规则与边界情况

- 只有已打开且有路径的文档可普通保存；内存新文档不得静默创建文件，保存入口应禁用或明确提示需要后续“另存为”能力。
- 未修改文档触发保存时不得写盘；可直接保持当前状态。
- 只读文档不得进入写入流程；后端仍必须校验只读状态，不能只依赖前端禁用按钮。
- 同一窗口同时只允许一个打开或保存流程。保存期间编辑器只读，并禁用打开与重复保存；完成或失败后恢复。
- 保存请求必须绑定后端已知的文档 ID。前端不得提交任意文件路径，也不得自行决定可信指纹；未知或过期 ID 必须拒绝。
- 后端应保存当前文档的路径、编码、换行、指纹和只读信息，并在每次成功保存后原子更新指纹与字节数，供连续保存使用。
- 成功只能对应本次提交的完整内容；在成功响应前不得清除未保存标记或修改会话描述信息。
- 任一保存错误都不得改变编辑器内容或清除未保存状态；临时文件和磁盘原文件的保护继续由 Rust 保存核心保证。
- 外部修改冲突必须拒绝覆盖并给出明确提示；强制覆盖与重新加载留待后续功能。
- Mixed 换行、GBK 不可编码内容或 `EncodingAmbiguous` 必须明确拒绝；提示可说明需要后续选择换行或另存为 UTF-8，但不得展示当前尚不存在的可执行操作。
- 错误使用稳定代码与必要的安全字段映射，不向前端或用户暴露内部路径、临时文件名或 Rust 调试文本。
- 前端不得因此获得宽泛文件系统、shell、dialog 或网络权限。

## 验收条件

- [ ] 已打开的 UTF-8、UTF-8 BOM 和可无损重开的 CP936 文件修改后可保存到原路径，磁盘内容、编码与换行正确。
- [ ] 保存成功后未保存标记清除，新的字节数和指纹用于下一次连续保存。
- [ ] 文本内容经二进制 IPC 传输；保存命令按后端文档 ID 解析目标，前端不能提交任意路径。
- [ ] 新建文档、未修改文档和只读文档不会被普通保存错误写入。
- [ ] 保存期间编辑器、打开入口和重复保存受到保护，结束后恢复可用。
- [ ] 大小超限、Mixed、不可编码、编码歧义、外部修改和一般写入失败均保留当前内容与未保存状态，并显示稳定、可理解的提示。
- [ ] 外部修改冲突不会覆盖磁盘变化；失败不会被误报为已保存。
- [ ] 前端未获得新增的宽泛文件系统、shell、dialog 或网络权限。
- [ ] 自动化测试覆盖后端状态绑定与错误映射、前端保存状态机及成功/失败会话更新；macOS 完成真实文件保存验收，Windows 待对应环境验证时明确记录。
