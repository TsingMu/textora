# 另存为与新建文档首次保存

> 状态：已完成；Windows 验证待对应环境执行

## 背景与目标

Textora 已能把已打开文件安全保存回原路径，但内存中的 Untitled 文档尚不能落盘，用户也无法为已有文档选择新路径、编码或换行格式。本功能交付下一个最小垂直切片：通过系统保存对话框取得可信目标，让新建文档可以首次保存，让已有文档可以安全另存，并在目标编码或换行无法沿用时提供明确选择。

## 范围

- 新建文档触发“保存”时进入首次保存流程；已有文档提供清晰可发现的“另存为”入口。
- 在打开系统保存对话框前提供保存格式选择：UTF-8（无 BOM）、UTF-8 BOM、严格 GBK/Windows CP936，以及 LF、CRLF。
- 新建文档默认 UTF-8（无 BOM）与 LF；已有文档默认沿用当前编码与单一换行格式。
- Mixed 文档另存时必须明确选择 LF 或 CRLF，不允许把 Mixed 作为目标格式。
- 文件路径只由 Rust 侧系统保存对话框取得；已有文档提交后端已知的文档 ID，Untitled 明确使用首次保存流程且不把前端本地 ID 冒充为可信 ID；两者只另行提交格式选择和二进制 UTF-8 内容。
- 复用 Rust 文档核心完成严格编码、50 MiB 限制、目标校验和安全落盘。
- 保存成功后返回新的文档描述信息，把当前会话关联到新目标并清除未保存标记；之后普通保存写入该目标。
- 取消或失败时保留当前内容、原有文档关联和未保存状态，并显示与保存阶段相符的提示。

## 非范围

- 普通保存冲突后的重新加载、强制覆盖或取消选择。
- 关闭未保存文档保护、多标签、多窗口、自动保存、备份和持续外部修改监听。
- 文件改名或移动的独立管理界面，以及文件夹选择、批量保存或导出流程。
- UTF-8、UTF-8 BOM、GBK/Windows CP936 之外的编码，以及 LF、CRLF 之外的目标换行格式。
- 文件菜单、系统菜单和键盘快捷键；本切片只要求可发现且可验收的界面入口。
- 持久化“GBK 编码提示”或改变 D-006 的严格无损重开规则。

## 用户流程

1. 用户在 Untitled 文档触发“保存”，或在已有文档触发“另存为”。
2. 应用展示格式选择；用户确认编码与换行，或取消并返回编辑。
3. Rust 侧打开系统保存对话框；用户选择目标文件，系统按平台惯例处理已有文件的覆盖确认，或取消并返回编辑。
4. 后端对已有文档按文档 ID 取得可信会话；对 Untitled 使用无既有路径的首次保存分支。随后验证内容和格式、检查目标状态并安全写入。
5. 成功时当前会话关联到新目标、更新描述信息并清除未保存标记；已有文档的原文件保持不变。失败时当前会话和磁盘原内容均保持可恢复状态。

## 行为规则与边界情况

- 新建文档的默认格式为 UTF-8（无 BOM）与 LF。已有文件另存时默认沿用其编码；原换行为 LF 或 CRLF 时默认沿用，原换行为 Mixed 时必须由用户选择 LF 或 CRLF。
- Untitled 当前只有前端本地 ID，后端不得把它当作已打开文档的可信 ID。首次保存成功后才创建后端文档关联并返回后端生成的 ID；已有文档另存成功后沿用其后端文档 ID，只原子更新关联目标和描述信息。
- 格式选择和系统保存对话框任一阶段取消，都不得创建、覆盖文件，改变文档关联，清除未保存标记或丢失编辑内容。
- 已有文档另存到不同目标时，不得修改原文件；成功后当前会话改为关联新目标，后续普通保存只写入新目标。
- 若用户选择当前文档的原路径，后端必须沿用普通保存的文档 ID、打开指纹和冲突保护，不得借“另存为”绕过外部修改检测。
- 后端能观测目标状态的最早时机是系统保存对话框**返回之后**。从 OS 覆盖确认到后端首次观测之间存在**不可关闭的窗口**，期间目标被创建、修改或删除都无法被检测；因此竞争保护从后端首次观测到的目标状态开始，且仅为 best-effort（与 `safe-save-core` 的 TOCTOU 处理一致），本切片不声称完全覆盖确认后的所有变化。
- 若对话框返回后后端首次观测到一个**已存在**的不同目标，记录该观测状态（指纹），并在替换前 best-effort 再次校验；**首次观测之后**到替换之前发生变化时拒绝覆盖。OS 覆盖确认只代表用户意图，不替代此后校验。
- 若对话框返回后后端首次观测到目标**不存在**，必须先在同目录写临时文件并 `sync_all`，再以**原子且不覆盖**的方式提交到目标（候选原语 `std::fs::hard_link(temp, target)`：Unix `link(2)` 与 Windows `CreateHardLinkW` 在目标已存在时均失败，从而既原子又不会覆盖已出现的新文件）；提交失败或写入/同步/进程异常时仅清理那个带唯一名称的临时文件，绝不按目标路径删除——避免在目标路径留下部分文件，也避免误删竞争产生的新文件。**首次观测之后**到提交之前被其他进程创建时，提交失败并保留该新文件。
- 所有编码、换行规范化和大小校验必须在目标发生破坏性变化之前完成。输出超过 50 MiB 时拒绝保存。
- GBK 使用严格无替换编码；不可表示字符必须返回字符与 UTF-8 字节偏移。按 GBK 写出后若无法被现有打开流程无损识别为 GBK，仍按 D-006 返回编码歧义；用户可改选 UTF-8 或 UTF-8 BOM 重试。
- UTF-8 BOM 输出必须恰好包含一个文件头 BOM；编辑内容中的 U+FEFF 不得被误删。LF/CRLF 选择只规范化行结束，不改变其他文本内容。
- 对符号链接目标的行为应与普通保存一致：保留链接本身并更新其真实目标；不得因另存为引入不同的链接处理语义。
- 已存在目标的权限不得被放宽；新目标使用操作系统和运行环境的默认权限。只读源文件仍可另存到其他可写目标。
- 内容继续使用二进制 IPC，不得变成大型 JSON 字符串或数字数组；前端不得提交任意目标路径、可信指纹或临时文件名。
- 同一窗口同时只允许一个打开或保存流程。格式选择、系统对话框和写入期间禁用打开、普通保存、另存为及重复触发；进入实际写入后编辑器只读，结束后恢复。
- 写入内容快照在应用内格式选择确认后、调用系统保存对话框前取得；从取得快照到系统对话框取消或后端响应期间编辑器必须只读，成功响应只能提交这份完整快照。系统对话框取消时丢弃快照，不改变当前会话。
- 后端只在安全写入完整成功后更新文档 ID 对应的路径、显示名、编码、换行、指纹、字节数和只读状态；失败响应不得提前改变可信会话。
- 错误使用稳定代码和必要的安全字段，不向前端或用户暴露内部路径、临时文件名或 Rust 调试文本。
- 前端不得因此获得宽泛文件系统、shell、dialog 或网络权限；系统保存对话框继续由受限 Rust 命令发起。

## 验收条件

- [x] Untitled 文档可用默认 UTF-8（无 BOM）与 LF 首次保存；成功后显示新文件信息、未保存标记清除，后续普通保存写入同一目标。
- [x] Untitled 文档可分别选择 UTF-8 BOM、GBK 和 CRLF；磁盘字节、BOM、编码与换行符合选择，严格 GBK 的歧义或不可编码内容会被拒绝。
- [x] 已有 UTF-8、UTF-8 BOM、可无损重开的 CP936、LF 和 CRLF 文件另存时默认沿用当前格式；另存成功后原文件不变、当前会话关联新目标。
- [x] Mixed 文档另存前必须明确选择 LF 或 CRLF，写出结果不再包含 Mixed 换行；未选择时不能进入写入。
- [x] GBK 不可表示字符和编码歧义均保留当前内容与未保存状态；改选 UTF-8 或 UTF-8 BOM 后可以安全重试。
- [x] 取消格式选择或系统保存对话框不会创建、覆盖文件，也不会改变内容、原文档关联或未保存状态。
- [x] 已存在目标按平台惯例取得覆盖确认；后端在对话框返回后首次观测目标并记录状态，首次观测之后到替换之前若目标被外部修改，保存拒绝且外部内容不被覆盖（OS 确认到首次观测之间的窗口不可关闭，见行为规则）。
- [x] 原本不存在的目标在对话框返回后首次观测时不存在；首次观测之后到提交之前若被其他进程创建，保存拒绝且新出现文件不被覆盖，提交采用原子且不覆盖的方式（不直接对目标 `create_new`，避免部分文件与误删）。
- [x] 选择当前原路径不会绕过普通保存的指纹、只读和冲突保护；符号链接行为与普通保存一致。
- [x] 大小超限、只读目标、权限失败和一般 I/O 错误均不会留下被误报为成功的会话，也不会破坏原目标。
- [x] 文本内容经二进制 IPC 传输；已有文档 ID 必须匹配后端可信状态，Untitled 仅在首次保存成功后取得后端生成的 ID；目标路径只存在于 Rust 可信状态，前端 capability 未新增宽泛文件系统、shell、dialog 或网络权限。
- [x] 格式选择、系统对话框和写入期间的操作互斥有效；成功只提交本次完整内容，失败后编辑器恢复可用。
- [x] 自动化测试覆盖首次保存、另存为、格式转换、取消、错误映射、会话提交以及目标竞争；交互式 macOS 真实文件交互验收已完成，Windows 待对应环境验证并在记录中明确。

## 实现前调查结论（2026-07-22）

**1. 系统保存对话框覆盖语义（`tauri-plugin-dialog` 2.7.2 / `rfd`）**

- API：`app.dialog().file()` 链式 `set_file_name` / `set_directory` / `add_filter` / `set_title` 后，`blocking_save_file() -> Option<FilePath>`（异步命令内可调用，与已用的 `blocking_pick_file` 一致）。`FilePath` 经 `.into_path()` 得 `PathBuf`。
- **无覆盖确认开关**：覆盖提示由 OS 原生处理（macOS `NSSavePanel`、Windows `IFileSaveDialog` 的 `FOS_OVERWRITEPROMPT`）。
- 返回值仅 `Option<FilePath>`：用户确认（含已存在目标的 OS 覆盖提示）后返回路径，取消返回 `None`；**不告知目标是否已存在或是否触发了覆盖提示**。
- 结论：原生对话框只能在**返回后**给出路径，无法捕获用户确认瞬间的目标状态。应用层必须在拿到路径后自行**首次观测**目标状态（是否存在/指纹），并在写入前做 best-effort 再校验；OS 覆盖确认只代表用户意图，不替代后端竞争保护。从 OS 确认到后端首次观测之间的窗口不可关闭，本切片不声称覆盖该窗口内的变化。

**2. 安全保存核心的最小扩展（无需新依赖；落盘与权限策略按下方修正）**

- 现状缺口：`document::save_document` 经 `canonicalize` 解析目标，目标不存在即返回 `SaveConflict`，**无法处理首次保存或另存到新目标**；且 `SaveRequest.read_only` 是源描述符标记，需区分普通保存与另存为后决定是否适用。
- 扩展方案：把目标语义抽象为 `SaveTarget`（或操作类型），由核心按类型执行不同规则：
  - `Replace { expected_fingerprint }`：目标须存在且匹配指纹。普通保存用打开指纹；另存到已存在目标用「对话框返回后首次观测到的指纹」。复用现有 `verify_no_conflict` + 同目录临时文件 + `rename` + 替换前 best-effort 再校验；保留目标权限。残留 TOCTOU 窗口（best-effort，与 `safe-save-core` 已记录一致）。
  - `Create`：目标须不存在。**不得**直接对目标使用 `OpenOptions::create_new` 当作安全原子落盘——写入、`sync_all` 或进程异常会在目标路径留下部分文件，按路径清理还可能误删竞争产生的新文件。改为：先在同目录写带唯一名称的临时文件并 `sync_all`，再以**原子且不覆盖**的方式提交到目标；候选原语 `std::fs::hard_link(temp, target)`（Unix `link(2)` 返回 `EEXIST`、Windows `CreateHardLinkW` 返回 `ERROR_FILE_EXISTS` 时即视为目标已存在而失败，从而既原子又绝不覆盖）。提交失败或异常仅清理那个唯一命名的临时文件，绝不按目标路径删除。OS 默认权限；首次观测之后到提交之前被他人创建则提交失败、保留新文件。
  - 实现期需在两平台验证 `hard_link` 的不覆盖语义；若目标文件系统不支持硬链接（本地 APFS/NTFS 均支持），应作为可理解的 I/O 失败暴露，绝不退化为可能覆盖的写法。同卷临时文件保证 `hard_link` 不跨文件系统。崩溃残留的唯一命名临时文件可由后续启动清理任务回收，本切片不要求。
- 源只读校验**保留在 Rust 保存核心内**（不外移到 IPC/前端，以遵守 `safe-save-core` 已确认的不变量「只读描述符必须拒绝普通保存」）：通过 `SaveTarget`/操作类型区分——普通保存（目标即源路径）由核心继续校验源描述符只读；另存为（目标为不同路径）才跳过源只读校验，使只读源可另存到其他可写目标。目标只读仍由核心 best-effort 保护。
- 符号链接：`Replace` 复用现有 `canonicalize(目标)`；`Create` 用 `canonicalize(父目录) + 文件名` 解析路径中的链接，临时文件与提交均作用于解析后的真实路径，保持与普通保存一致的链接行为（保留链接、更新真实目标）。
- 路由：IPC 检测「所选路径规范化后等于当前文档原路径」时走普通保存（沿用打开指纹、源只读校验与冲突保护），不借另存绕过外部修改检测。
- 结论：标准库（`rename` + `hard_link` + `create_new` 临时文件）能满足两平台「已存在目标安全替换」与「不存在目标不覆盖原子提交」；**不引入新依赖**。`Create` 的不覆盖原子性来自 `hard_link`（非 `create_new` 直接落盘）。

**3. 格式选择界面**

- 采用应用内轻量界面（编码：UTF-8 / UTF-8 BOM / GBK；换行：LF / CRLF），在系统保存对话框**之前**串行呈现。默认值：Untitled → UTF-8（无 BOM）+ LF；已有文档 → 沿用当前编码，源换行为 LF/CRLF 时沿用、为 Mixed 时强制选择。取消任一阶段都不落盘、不改文档关联、不清未保存标记。具体视觉布局不改变本规格定义的顺序、默认值与取消语义。

实现要点：

- **保存核心扩展（`document/save.rs`）**：新增 `SaveTarget { InPlace { source_read_only, expected }, ExistingTarget { observed }, NewTarget }`。源只读校验保留在核心且仅对 `InPlace` 生效（遵守 `safe-save-core` 不变量）；另存为（`ExistingTarget`/`NewTarget`）不阻止只读源。`NewTarget` 按「调查结论」以同目录临时文件 + `sync_all` + `std::fs::hard_link(temp, target)` 原子且不覆盖提交（Unix `EEXIST` / Windows `ERROR_FILE_EXISTS` 即 `SaveConflict`），异常仅清理带唯一名称的临时文件，绝不按目标路径删除。
