# 保存冲突解决

> 状态：实现中

## 背景与目标

Textora 已能在普通保存前检测文件是否自打开或上次成功保存后被外部修改，并会拒绝覆盖，但用户目前只能看到错误，无法在应用内继续处理；文件被外部删除时也不会在用户返回应用时主动说明。本功能交付一个最小、完整的冲突解决切片：确认磁盘文件仍存在但内容已经变化时，用户可以重新加载磁盘版本、明确强制覆盖或取消；应用重新获得焦点并发现文件已被删除时，询问用户是否把当前内容保留在应用中。任何选择失败都不得丢失当前编辑内容或误报保存成功。

## 范围

- 普通保存因当前关联文件被外部修改而发生冲突时，展示应用内冲突解决界面。
- 对仍然存在且可安全观测的目标提供“重新加载”“强制覆盖”“取消”三个明确操作。
- 重新加载通过 Rust 侧可信文档路径读取一致快照；完整成功后才以磁盘内容替换当前编辑内容并清除未保存状态。
- 强制覆盖在用户明确选择后，以当前冲突对应的完整编辑快照覆盖最新观测到的磁盘版本，并继续复用严格编码、大小限制、只读保护、竞争复核与原子保存。
- 应用窗口从失焦变为聚焦时，通过 Rust 侧可信文档路径检查当前关联文件是否仍存在；检查不得读取或上传编辑内容。
- 文件已经不存在时询问用户是否在应用中保留当前内容：选择“保留”后解除原路径关联并保留为需要另存的内存文档；选择“不保留”后关闭该文档。
- 内容变化冲突中选择取消，或重新加载/强制覆盖失败时，保留编辑内容、文档关联、编码、换行和未保存状态。

## 非范围

- 文本合并、差异比较、冲突副本、备份、版本历史或恢复已被用户明确放弃的内容。
- 持续监听外部修改、定时轮询、自动重新加载或自动保存；本切片只在应用窗口重新获得焦点时检查当前文件是否缺失。
- 未保存关闭保护、多标签、多窗口，以及由关闭流程触发的保存冲突协调。
- 为缺失文件增加原路径“重新创建”操作，或自动打开系统保存对话框。
- 处理普通保存之外的另存为目标竞争；另存为仍按 `save-as-and-first-save.md` 的安全拒绝语义处理。
- 放宽只读文件、编码、换行、50 MiB 上限、最小 capability 或 best-effort TOCTOU 边界。

## 用户流程

1. 用户编辑已打开文件并触发普通保存。
2. Rust 保存核心发现当前磁盘内容与后端保存的可信指纹不一致，拒绝写入；应用展示冲突解决界面，并说明两份内容的来源和各操作后果。
3. 用户选择：
   - **重新加载**：放弃当前未保存编辑，读取并显示最新磁盘内容。
   - **强制覆盖**：明确放弃已观测到的外部修改，用当前编辑快照保存。
   - **取消**：不读写磁盘，返回编辑。
4. 重新加载或强制覆盖完整成功后，应用更新可信文档描述信息并结束冲突；失败或再次发生竞争时继续保留当前编辑内容，并允许用户重试或取消。
5. 用户离开应用后，当前文件被外部删除；用户再次聚焦应用窗口时，后端检查可信路径并返回“文件已缺失”。
6. 应用询问是否把文件保留在应用中：
   - **保留**：保留当前编辑内容、编码、换行和可识别的文件名，解除已删除路径的可信关联，并把文档标为未保存；下次保存进入首次保存/另存为流程。
   - **不保留**：明确放弃当前内存内容并关闭文档；当前单文档界面回到新的空白文档。
7. 若普通保存早于焦点检查发现文件已缺失，也进入同一个“保留或关闭”流程，不显示重新加载或强制覆盖操作。

## 行为规则与边界情况

- 内容变化冲突由普通保存返回的稳定冲突结果触发；文件缺失既可由重新聚焦检查发现，也可由普通保存发现。后端必须区分至少“内容已变化”和“目标已缺失”，不能仅凭前端文案或前端文件状态猜测原因。
- 重新聚焦检查仅针对后端仍持有可信路径的当前文档，并绑定当前活动文档 ID；Untitled、已经选择保留的内存文档、正在打开/保存/另存为或已有提示待处理时不得重复检查或弹出第二个提示。
- 焦点事件可能频繁或由系统文件对话框间接产生。相同文档同一轮聚焦只允许一个缺失检查；过期检查结果不得覆盖已经打开、关闭、另存或切换后的会话状态。
- 聚焦检查只判断可信目标是否缺失。文件仍存在但内容已经变化时，本切片不在聚焦时提示或自动读取，继续由普通保存的指纹检查发现并处理。
- 重新加载与强制覆盖请求必须绑定后端当前活动文档 ID 及其仍有效的冲突状态。前端不得提交任意路径、可信文件指纹、临时文件名或自行声明冲突类型；未知、过期或已经解决的请求必须在文件 I/O 前拒绝。
- 普通保存首次返回冲突时不得更新后端保存指纹、字节数或前端文档描述信息，也不得清除未保存标记。
- 冲突界面必须明确说明“重新加载会放弃当前未保存编辑”和“强制覆盖会放弃磁盘上的外部修改”。选择对应操作本身即为明确确认，不得由默认焦点或关闭界面隐式触发破坏性操作。
- 冲突待处理期间保留导致冲突的完整编辑快照，编辑器只读，并禁用打开、普通保存、另存为及重复解决操作；取消后恢复编辑。文件缺失提示待处理期间同样禁止并发文件操作。
- 重新加载必须从后端可信路径建立一致快照，并继续执行 50 MiB 限制、严格编码检测和读取期间变化检测。只有元数据与完整二进制内容均成功取得后，才能原子替换前端会话内容、更新后端可信描述信息并清除未保存状态。
- 重新加载失败（包括读取期间再次变化、文件消失、超限、编码无效、权限或一般 I/O 错误）不得提前替换编辑器内容或后端活动文档；冲突界面保持可处理状态，并显示稳定、可理解的错误。
- 强制覆盖不是关闭所有保护。后端在用户确认后必须重新观测可信目标，以该最新观测结果作为本次覆盖基线，再执行现有严格编码、换行规范化、50 MiB 限制、只读/权限检查、同目录临时文件和原子替换；前端不能提供或修改这条基线。
- 强制覆盖从“确认后的最新观测”到最终替换之间仍须进行 best-effort 冲突复核。期间目标再次变化或消失时必须再次拒绝，而不是覆盖更新后的内容；用户仍保有当前编辑快照并可重新选择。
- 强制覆盖不得绕过源或目标的当前只读状态。目标只读、权限失败、内容不可编码、编码歧义、Mixed 未选择换行、大小超限或一般 I/O 失败时均不得修改磁盘目标或清除未保存状态。
- 重新加载成功后，编码、换行、字节数、指纹、显示名和只读状态以新磁盘快照为准；文档继续使用当前后端 ID 和可信用户选择路径。符号链接继续保留链接本身，并读取或覆盖其真实目标。
- 强制覆盖成功后，编辑内容、编码和换行保持本次普通保存的语义，后端更新最终字节数、指纹和当前只读状态；随后连续普通保存使用新的可信指纹。
- 文件缺失时不得把“强制覆盖”解释为创建新文件，因为缺失目标没有可确认放弃的外部版本。应用改为询问“是否在应用中保留”，不得自动进入另存为或重建原路径。
- 选择“保留”必须先完整保留当前内存内容、编码、换行和便于识别的原文件名，再解除前后端对已删除路径的可信关联并将文档标为未保存；即使删除前文档没有本地编辑，也必须标为未保存。此后普通保存进入既有首次保存/另存为流程，不能写回已删除路径。
- 选择“不保留”是明确的破坏性操作：若文档含未保存编辑，提示必须说明这些编辑也会被放弃；确认后关闭当前文档并清理其后端可信状态。当前单文档界面创建新的空白文档，不能继续保留已关闭文档的内容、路径或保存错误。
- 关闭文件缺失提示或按 Escape 等价于“保留”，不能隐式执行“不保留”。提示期间窗口再次失焦不代表任何选择，提示保持待处理并在重新聚焦后继续显示；无法确认用户选择时同样保持内容和提示，不关闭文档。
- 内容变化冲突中选择取消不得读取、创建或覆盖文件，不得改变内容、文档关联、可信指纹、描述信息或未保存状态；仅退出冲突状态并恢复编辑与文件操作。
- 关闭内容变化冲突界面、按 Escape 或系统级取消均等价于“取消”，不能默认执行重新加载或强制覆盖。
- 错误通过稳定代码和必要的安全字段传递，不向前端或用户暴露内部路径、临时文件名、文件指纹、冲突令牌或 Rust 调试文本。
- 内容继续使用二进制 IPC；前端 capability 不新增宽泛文件系统、shell、dialog 或网络权限。
- 与既有保存核心相同，跨平台文件系统无法提供按指纹的严格原子 compare-and-swap；规格只要求尽可能晚的再次校验和原子替换，不宣称完全关闭最后一次校验与替换之间的 TOCTOU 窗口。

## 验收条件

- [ ] 已修改文档的磁盘文件被外部改写后，普通保存不会覆盖外部内容，并展示重新加载、强制覆盖和取消操作。
- [ ] 取消冲突后，编辑内容、未保存状态、文档关联和磁盘内容均保持不变，编辑器恢复可用。
- [ ] 重新加载成功后显示完整的最新磁盘内容，清除未保存状态，并以新编码、换行、字节数、指纹和只读信息支持后续编辑与保存。
- [ ] 重新加载遇到文件再次变化、消失、超限、无效编码、权限或 I/O 失败时，当前编辑内容和未保存状态保持不变，失败不会被误报为已加载。
- [ ] 强制覆盖成功后磁盘内容等于冲突时保留的完整编辑快照，外部版本被明确替换，未保存状态清除，随后普通保存可继续工作。
- [ ] 强制覆盖确认后、最终替换前若目标再次变化或消失，本次覆盖再次拒绝，最新外部内容与当前编辑快照均得到保留。
- [ ] 强制覆盖不能绕过只读、权限、编码、换行和 50 MiB 限制；失败不留下部分文件，也不错误更新会话状态。
- [ ] 应用窗口失焦期间当前文件被外部删除后，重新聚焦会且只会显示一次“是否在应用中保留”提示；Untitled、忙碌状态和过期检查结果不会误触发或覆盖新会话。
- [ ] 文件缺失时选择“保留”会完整保留当前内容与格式，解除原路径关联并标记为未保存；随后保存进入首次保存/另存为流程，不会在原路径静默重建。
- [ ] 文件缺失时选择“不保留”会在明确告知未保存编辑将被放弃后关闭文档，并回到新的空白文档；磁盘上不会创建或修改文件。
- [ ] 文件缺失提示被关闭或按 Escape 时按“保留”处理；提示期间再次失焦或无法取得明确选择时保持原内容和待处理提示，不会误关闭或重复创建提示。
- [ ] 符号链接冲突解决保留链接本身，并对真实目标执行重新加载或强制覆盖；后续普通保存仍关联用户原先选择的路径。
- [ ] 未知、过期或已解决的文档/冲突请求在文件 I/O 前被拒绝；前端不能指定任意路径、指纹或冲突类型。
- [ ] 冲突或文件缺失提示期间的编辑和文件操作互斥、重复点击保护、Escape/关闭安全默认语义有效；成功、失败、取消、保留和关闭后的界面状态一致。
- [ ] 自动化测试覆盖后端冲突分类、可信状态提交与清理、焦点检查的竞态/去重、重新加载和强制覆盖的成功/失败/再次竞争，以及前端提示状态机与操作互斥；macOS 13+ 与 Windows 10 22H2+ 的真实文件交互分别验证并如实记录。

## 依赖与约束

- 属于 `docs/features/basic-text-editing.md` 的外部修改保护子切片。
- 依赖已完成的 `docs/features/safe-save-core.md`、`docs/features/save-opened-file.md` 与 `docs/features/save-as-and-first-save.md`；不得在前端重复实现读取、编码、冲突检测或安全落盘。
- 遵守 `docs/ARCHITECTURE.md` 的 Rust 文档核心、后端可信文档状态、二进制 IPC、失败保护和最小权限不变量。
- 编码规则由 D-005 与 D-006 约束，平台和技术边界由 D-003 与 D-004 约束。
- 本功能完成后才适合把保存行为接入未保存关闭保护；关闭保护与本功能完成后再进入多标签会话。

## 开放问题

暂无会阻止本切片确认或实现的问题。文件缺失时“保留”表示保留应用内内容并解除原路径关联，“不保留”表示明确关闭；若未来需要在原路径重新创建已删除文件，应作为独立、明确命名且带破坏性确认的行为另行规格化。
