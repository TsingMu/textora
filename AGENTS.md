# Textora AI Working Agreement

本项目采用轻量的 **SDD + Project Memory** 工作流，面向个人开发及 Codex、Claude、ChatGPT 等 AI 编程工具。

## 开始工作前

根据任务范围阅读以下文件：

- `README.md`：项目入口、运行方式和协作约定
- `docs/PRODUCT.md`：产品定位、用户、范围、非目标和产品原则
- `docs/ARCHITECTURE.md`：系统结构、模块边界和架构不变量
- `docs/DECISIONS.md`：重要且尚未被取代的产品与技术决策
- 相关的 `docs/features/*.md`：功能需求、边界情况和验收条件
- `docs/tasks/current.md`：当前任务、进度和会话交接信息

若文件尚不存在，应根据当前任务判断是否需要创建，不要为了形式机械补齐全部文档。

## 文档职责

- `PRODUCT.md` 记录稳定的产品方向和跨功能规则。
- `ARCHITECTURE.md` 记录技术边界、数据流和不能破坏的约束。
- `DECISIONS.md` 只记录重要、非显然、难以撤销或容易被重复质疑的决策。
- Feature Spec 定义单个功能“做什么、什么不做、如何验收”。
- `current.md` 只保留当前少量可执行任务，个人项目同时最多一个任务处于进行中。
- `backlog.md` 保存尚未承诺实施的想法，不代表开发计划。

标准流程：

```text
Backlog → Feature Spec → Current Tasks → Implementation → Verification
```

## 实现规则

1. 不静默创造、改变或扩大需求；存在歧义时明确说明假设。
2. 遵守现有产品规则、架构边界和已接受的历史决策。
3. 只修改当前任务及其必要依赖，不重构无关代码。
4. 优先完成最小但完整、可验证的垂直切片。
5. 不无故引入新依赖、架构层或抽象。
6. 根据风险考虑加载、空状态、错误、权限、取消、恢复及破坏性操作保护。
7. 文档与代码冲突时应报告冲突和影响，不得静默选择或改写其中一方。

## 完成与验证

任务完成前：

- 对照 Feature Spec 验证相关验收条件。
- 运行适当的测试、构建、格式化或静态检查。
- 只报告实际执行过的验证；未运行的检查不得描述为通过。
- 更新 `docs/tasks/current.md` 中的状态和必要的会话交接信息。
- 仅在约定行为发生变化时更新 Feature Spec。
- 仅在产品方向、架构边界或重要决策发生变化时更新对应长期文档。

## Git 提交约定

- Git 提交标题与正文统一使用简体中文。
- 代码标识符、文件名、命令、库名和无法准确翻译的技术术语可保留原文。
- 提交信息应简洁说明实际变更，不记录未执行的验证或未完成的工作。

## 文档维护原则

只记录未来的开发者或 AI 无法轻易从代码和 Git 历史中恢复的信息，避免重复维护同一内容。

保留：产品意图、跨功能规则、架构边界、关键决策、验收条件和当前真实状态。

通常不记录：显而易见的实现细节、普通命名选择、短期讨论、流水账以及无执行计划的详细设计。

判断标准：

> 这条信息是在帮助未来的我和 AI，还是只是在重复代码或满足流程？
